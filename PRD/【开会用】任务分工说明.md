### 开发策略：硬编码先行

**第一周（现在）：假数据直接写死在组件里，先把 UI 跑通。**  
**第三周（5.11-5.17）：后端 API 就绪后，统一替换成真实 GraphQL 数据。**

这样你不会被后端进度卡住，第一周就能看到完整页面效果。

### 注意事项

1. **第一阶段：假数据可以写死**  
   `const members = [{ name: '张三', title: '主席', ... }]` 直接写在组件里没问题。**不需要**抽成变量或 props，等第三周替换时再整理。

2. **严格遵守 Design Token**  
   主色 `#BA1C21`，背景 `#FAF8F5`，文字 `#2A2424`，全部用 CSS 变量，不要自己写颜色值。

3. **先做 Desktop，再做 Mobile**  
   每个页面完成 Desktop 布局后，立即检查 Mobile（375px）是否正常，不要最后统一做适配。

4. **Join Us 页的表单：第一阶段用固定字段**  
   `join/page.tsx` 先硬编码三个固定字段（姓名、邮箱、年级），第三周再替换成动态 `form_schema` 渲染。

5. **Framer Motion 动效不要过度**  
   `AnimatedSection` 用于版块进入动效即可，不要每个元素都加动画，避免页面卡顿。

### 注意事项

1. **SchemaBuilder 是最难的部分，最优先开始**  
   动态表单构造器（管理员可以自定义报名表单的字段）逻辑最复杂，不要留到最后。第一周就开始做原型。

2. **所有表单必须接 Zod 校验**  
   用 React Hook Form + Zod 的组合。不能用原生 HTML form 直接 submit，必须走 schema 校验。

3. **数据请求全部走 TanStack Query**  
   所有接口调用都要通过 `useQuery` / `useMutation`，不能裸写 `fetch`。好处是自动缓存、自动 loading 状态。

4. **第一阶段用 Mock 数据，第三周接真实 API**  
   第一周 Mock 数据跑通 UI 和交互逻辑，第三周（5.11 起）Mike 后端就绪后统一替换为真实 Strapi API 调用。

5. **后台不对外公开，部署在不同端口**  
   本地开发跑在 `localhost:3001`，和客户端前端（`localhost:3000`）不一样。

### 注意事项

1. **第一周优先把 Strapi 和 GraphQL 跑通**  
   Emilia 和 Yeon 需要有 API 才能联调，后端越早跑通越好。目标：第一周结束前，所有人都能用 GraphQL playground 查到数据。

2. **`form_schema` 字段是关键**  
   `Events` 表里的 `form_schema` 是一个 JSON 字段，存的是报名表单的字段定义（如「需要哪些字段、类型是什么」）。Yeon 的 SchemaBuilder 生成的结果，存进这个字段。Emilia 的 Join Us 页读取这个字段渲染表单。三个人都依赖它，格式必须提前对齐。

3. **确认 `form_schema` 的 JSON 格式（第一周开会就定）**  
   建议格式：
   ```json
   [
     { "key": "name", "label": "姓名", "type": "text", "required": true },
     { "key": "email", "label": "邮箱", "type": "email", "required": true },
     { "key": "year", "label": "年级", "type": "select", "options": ["大一","大二","大三"], "required": false }
   ]
   ```
   这个格式一旦确定就不能随意改，改了会影响 Emilia 和 Yeon 的代码。

4. **Rate Limit 默认配置**  
   同一 IP 每分钟最多发 5 次报名请求，超过返回 429 状态码。

5. **本地开发用 Docker 启动数据库**  
   所有人本地开发必须用 `docker-compose up -d` 启动 PostgreSQL 和 Redis，不要本地直接装数据库。

6. **Strapi 管理员账号统一**

## 所有人必须遵守的规范与规则

### 1. 分支管理规范
1. **禁止直接向 `main` 或 `dev` 提交代码**，只能在自己的 `feature` 分支工作。
2. **分支命名格式：** 
   - 功能开发：`feature-你的名字-功能模块`，例如 `feature-emilia-首页`。
   - 文档修改：统一提交到 `docs` 分支（基于 `main` 分支拉取，例如 `docs-你的名字-更新说明` 或直接在公共 `docs` 协作），以后所有和文档相关的修改必须走文档专用分支。

### 2. Commit Message (提交信息) 规范（建议接近 Conventional Commits）
- **格式要求：** `<type>: <short summary>` (冒号后面加一个空格)
- **描述语言要求：** `<short summary>` 部分必须使用**中文**说明做了什么修改。
- **常用 type 参考：**
  - `feat`: 新功能 (feature)
  - `fix`: 修 bug
  - `refactor`: 重构（不改变功能）
  - `docs`: 文档修改
  - `test`: 测试相关
  - `chore`: 构建、依赖、脚本等杂项
  - `style`: 格式调整（空格、换行、linter 等，不影响代码运行）
- **示例：** `feat: 完成首页精选活动卡片组件` 或 `docs: 更新后台前端配色规范`

### 3. Pull Request (PR) 格式规范
- **语言要求：必须使用中文。**
- **发 PR 时写清楚你做了什么**，PR 描述区必须包含以下内容：
  1. **背景/目的：** 为什么做这个改动？解决什么问题？
  2. **修改内容：** 简要列出主要修改了哪些功能/文件。
  3. **测试情况（可选 / 涉及代码）：** 你在本地怎么测试的？贴一下截图或录屏。
- **注意：** 负责人看不懂、描述敷衍的 PR 会被直接打回。

### 4. 其他日常规范
3. **每次 commit 只做一件事**，不要积累一周再一次提交几百个文件。
5. **联调前提前通知**：Emilia 和 Yeon 在替换假数据之前，确认 Austin/Mike 的 API 已经就绪。
6. **遇到阻塞立刻在群里说**，不要自己卡一天没有进展，及时寻求帮助。
