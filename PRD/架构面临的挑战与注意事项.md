# 现架构面临的挑战与注意事项 (Trade-offs & Challenges)

虽然 **Strapi + NestJS** 结合是兼顾“开发效率”和“系统能力”的极佳选择，但本设计模式在工程实践中需要注意以下挑战：

### 1. 运维部署成本增加
* **双服务维护**：需要同时部署和维护两个独立的 Node.js 服务实例（内容网关 Strapi + 业务微服务 NestJS）。
* **依赖项增多**：还需搭配维护数据库（PostgreSQL）和缓存中间件（Redis，供 BullMQ 队列使用），增加了 CI/CD 及线上容灾的实现复杂度。

### 2. 数据交互与最终一致性
* 当 NestJS 微服务独立处理完高并发抢票 (`registration`) 日志或状态流转后，需要通过 API 调用或事件通知的形式，将最终数据变动同步回 Strapi 的底层数据库。
* 必须设计好两套服务间的容错重试机制（例如队列失败重载），从而保证数据的一致性，防止超卖或状态掉链。

### 3. 网关切分与请求路由策略
* 前端在发起请求时，必须具有严谨的路由分发设计：
  * **走 Strapi**：普通的静态/半静态数据浏览（如浏览活动列表、社员列表、赞助商信息等）。
  * **走 NestJS**：带有写操作、事务特征的高并发入口（如点击“立即报名”、支付校验等）。
* 可在 Nginx 反向代理层配置统一网关 (API Gateway / BFF)，或在前端侧的 API Client 中配置多 `BaseURL` 做好严格分类映射。
