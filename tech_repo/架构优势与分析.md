# 架构设计优势与分析 (Strapi + NestJS)

本项目后端采用了 **Strapi (Headless CMS) + NestJS (微服务)** 的组合，这在实际工程中是一种非常典型的“内容管理与复杂业务解耦”的设计模式，具有极高的合理性：

### 1. 明确的职责划分（各司其职）
* **Strapi 的角色（内容管理底座）：**
  * **优势**：Strapi 提供了开箱即用的后台管理界面（Admin UI）和基础的增删改查（CRUD）API。
  * **适用场景**：非常适合用来管理 CSSA 这种社团网站的静态/半静态数据，比如：文章发布、赞助商信息 (sponsors)、部门介绍 (departments)、过往活动展示 (past-events) 等。不需要写一行后端代码就能把基础数据接口配好。
* **NestJS 的角色（复杂业务微服务）：**
  * **优势**：处理高度定制化、高并发或需要后台异步处理的任务。
  * **适用场景**：例如活动报名注册 (registration) 和速率限制。活动报名往往涉及高并发抢票、表单校验等，这些是 Strapi 原生不擅长的。用 NestJS 来承载这些业务逻辑（含 Redis Rate Limit Guard），不仅稳定而且可扩展。

### 2. 团队协作上的优势
* **前端团队**可以直接对着 Strapi 生成的 GraphQL/REST API 并行开发数据展示页面，不需要等后端写好接口。
* **后端团队**则可以把精力集中在 NestJS 里写最有技术含量的核心逻辑（验证、排队、事务），而不用去写琐碎的基础增删改查。

### 3. 可扩展性与系统的安全性
* **性能**：如果遇到大型活动报名，可以只针对 NestJS 服务进行扩容，而不影响 Strapi 所在的服务器。
* **安全性**：独立的身份验证模块 (auth) 可以在 NestJS 中做更严格的拦截和鉴权。
